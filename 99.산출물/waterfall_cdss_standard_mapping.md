# CDSS 예시 기준 폭포수 표준산출물 정리

기준 사례:

- 대상 시스템: 병원행정시스템(HIS) 연계 CDSS
- 기능 1: 당뇨 환자 추적검사 누락 알림
- 기능 2: 패혈증 위험 알림

정리 기준:

- 현재 `20.산출물` 폴더에 있는 표준산출물 코드 체계(`REQ`, `ANA`, `DES`, `IMP`, `TES`, `TRA`)를 그대로 따름
- 각 산출물에 대해 `이 문서가 CDSS 프로젝트에서 무엇을 담아야 하는지`를 예시 중심으로 정리함

## 1. 분석 단계

| 코드 | 표준산출물 | CDSS 프로젝트에서 담아야 할 핵심 내용 | 예시 |
|---|---|---|---|
| `REQ01` | 아키텍쳐정의서 | HIS, EMR, LIS, 간호기록, CDSS 엔진, 알림 UI, 로그 저장소 간 전체 구조 | 외래 진료실 화면에서 HbA1c 누락 배너 표시, 응급실 화면에서 패혈증 고위험 팝업 표시 |
| `REQ02` | 사용자 요구사항 정의서 | 의사, 간호사, 코디네이터, 감염관리실, 운영자의 요구사항 | 의사는 진료 중 검사 누락을 보고 싶고, 감염관리실은 패혈증 경고 후 조치율을 보고 싶음 |
| `REQ03` | 용어사전 | 의료/시스템 용어 표준화 | HbA1c, UACR, qSOFA, bundle, override, alert fatigue |

### 분석 단계 예시 포인트

- 당뇨 알림은 `지속관리형 업무지원`
- 패혈증 알림은 `실시간 환자안전 지원`
- 두 기능 모두 `왜 알림이 떴는지 설명가능성`이 필요

## 2. 설계 단계 - 기본설계

| 코드 | 표준산출물 | CDSS 프로젝트에서 담아야 할 핵심 내용 | 예시 |
|---|---|---|---|
| `ANA01` | 업무정의서 | AS-IS / TO-BE 업무 흐름, 사용자 행동, 예외 절차 | 현재는 누락검사를 사람이 기억해야 하나, TO-BE에서는 진료화면에서 자동 경고 |
| `ANA02` | 유스케이스 명세서 | 사용자별 시나리오와 시스템 반응 | 주치의가 환자 차트를 열면 HbA1c 누락 여부를 판정하고 알림 표시 |
| `ANA03` | 요구사항 추적표 | 요구사항과 설계/구현/테스트 연결 | `HbA1c 누락 알림` 요구사항이 UI, 규칙, 테스트케이스, KPI로 연결됨 |
| `ANA04` | 엔티티 관계 모형 정의서 | 환자, 방문, 진단, 검사, 처방, 알림, 로그 간 관계 | 환자 1명은 여러 방문을 가지며, 방문마다 여러 알림 로그가 생성됨 |
| `ANA05` | 사용자 인터페이스 정의서 | 알림 위치, 문구, 상세보기, 무시사유, 후속버튼 정의 | 당뇨는 배너, 패혈증 고위험은 팝업, 상세에 발생 근거 표시 |
| `ANA06` | 인터페이스 정의서 | HIS/EMR/LIS/간호기록과 CDSS 연계 방식 | LIS에서 HbA1c 결과 수신, 활력징후 저장 이벤트로 qSOFA 재계산 |

## 3. 설계 단계 - 상세설계

| 코드 | 표준산출물 | CDSS 프로젝트에서 담아야 할 핵심 내용 | 예시 |
|---|---|---|---|
| `DES01` | 아키텍쳐 설계서 | 서비스 구성, 연계 방식, 장애 시 대체 흐름 | CDSS 장애 시 HIS 진료는 유지하고 경고만 비활성화 |
| `DES02` | 클래스 설계서 | 규칙 엔진, 판정 서비스, 로그 서비스 등 내부 객체 구조 | `DiabetesFollowupRuleService`, `SepsisRiskEvaluator`, `AlertLogService` |
| `DES03` | 컴포넌트 설계서 | 기능 단위 구성 요소와 책임 분리 | 환자조회 컴포넌트, 규칙판정 컴포넌트, 알림표시 컴포넌트 |
| `DES04` | 엔티티 관계 모형 설계서 | 논리 ERD를 물리 모델 수준으로 확장 | ALERT_LOG, ALERT_RULE_VERSION, PATIENT_VISIT, LAB_RESULT |
| `DES05` | 데이터베이스 설계서 | 테이블/컬럼/키/인덱스 정의 | 알림유형, 발생시각, 사용자ID, 처리결과, 무시사유 컬럼 |
| `DES06` | 사용자 인터페이스 설계서 | 화면별 와이어프레임과 동작 흐름 | 진료실 상단 당뇨 배너, 응급실 우선순위 팝업, 상세 팝업 |
| `DES07` | 인터페이스 설계서 | 메시지/배치/API 명세 | 활력징후 저장 시점 이벤트, 검사결과 HL7 수신 명세 |
| `DES08` | 데이터 전환 및 초기데이터 설계서 | 규칙 초기값, 코드 매핑, 기준 데이터 적재 | HbA1c 검사코드 목록, qSOFA 기준값, 당뇨약 코드셋 |
| `DES09` | 총괄시험 계획서 | 전체 시험 범위, 역할, 일정, 판정 기준 | 단위/통합/시스템/인수시험 범위와 임상 검증 절차 |
| `DES10` | 단위시험 케이스 | 함수/모듈 단위 테스트 정의 | 최근 HbA1c가 91일 전이면 알림 발생 |
| `DES11` | 통합시험 시나리오 | 연계 중심 시나리오 | LIS 결과 수신 후 알림 상태 자동 종료 |
| `DES12` | 시스템시험 시나리오 | 사용자 관점 전체 흐름 시험 | 응급실 접수 -> 활력 입력 -> 패혈증 고위험 팝업 -> 후속 오더 기록 |

## 4. 구현 단계

| 코드 | 표준산출물 | CDSS 프로젝트에서 담아야 할 핵심 내용 | 예시 |
|---|---|---|---|
| `IMP01` | 데이터베이스 테이블 | 실제 생성된 테이블 정의서 | ALERT_LOG, ALERT_TARGET_PATIENT, RULE_CONFIG |
| `IMP02` | 프로그램 코드 | 구현 모듈 목록과 핵심 로직 설명 | 당뇨 추적검사 판정 로직, qSOFA 계산 로직, UI 표시 로직 |
| `IMP03` | 단위시험 결과서 | 단위시험 실행 결과, 결함, 수정 내역 | HbA1c 90일 계산 오류 수정 후 재시험 통과 |
| `IMP04` | 운영전환 계획서 | 배포 절차, 사용자 교육, 모니터링 계획 | 외래 먼저 오픈 후 응급실 확대, 오픈 첫 2주 집중 모니터링 |

## 5. 시험 및 이행 단계

| 코드 | 표준산출물 | CDSS 프로젝트에서 담아야 할 핵심 내용 | 예시 |
|---|---|---|---|
| `TES01` | 통합시험 결과서 | 인터페이스/데이터 연계 시험 결과 | 활력징후 이벤트 누락 없이 qSOFA 재계산 확인 |
| `TES02` | 시스템시험 결과서 | 사용자 시나리오 시험 결과 | 외래 진료 시 당뇨 배너, 응급실 경고 팝업 정상 동작 |
| `TRA01` | 사용자 지침서 | 의사/간호사/코디네이터 사용법 | 알림 의미, 확인 방법, 무시사유 입력법 |
| `TRA02` | 운영자 지침서 | 운영팀/관리자 관리 방법 | 임계값 조정, 규칙 비활성화, 로그 조회 |
| `TRA03` | 시스템 설치 결과서 | 설치 및 배포 결과 | 운영 서버/테스트 서버 설치 내역 |
| `TRA04` | 운영전환 결과서 | 실제 오픈 결과와 이슈 정리 | 첫 주 오경보 사례, 문구 조정 여부 |
| `TRA05` | 데이터이행 결과서 | 기준 데이터와 코드셋 이행 결과 | 당뇨약 코드셋, 검사코드, 알림유형 초기값 적재 완료 |
| `TRA06` | 인수시험 시나리오 | 발주/사용자 관점 최종 시나리오 | 내분비내과와 응급의학과가 승인하는 최종 업무 시나리오 |
| `TRA07` | 인수시험 결과서 | 인수시험 수행 결과와 승인 | 당뇨 알림 승인, 패혈증 알림은 야간 문구 조정 조건부 승인 |

## 6. 이 표준체계에서 특히 중요한 산출물

CDSS 기준으로는 아래 문서들의 중요도가 높습니다.

1. `REQ02 사용자 요구사항 정의서`
   임상의/간호사/운영자의 실제 요구를 잘못 잡으면 알림이 바로 거부됩니다.
2. `ANA02 유스케이스 명세서`
   알림이 어느 시점에 누구에게 어떻게 떠야 하는지 구체화해야 합니다.
3. `ANA06 인터페이스 정의서`
   HIS, LIS, 간호기록 연계가 흔들리면 CDSS는 동작하지 않습니다.
4. `DES06 사용자 인터페이스 설계서`
   alert fatigue를 줄이려면 UI 설계가 중요합니다.
5. `DES09~DES12 시험계획/시나리오`
   의료 시스템은 기능 테스트만 아니라 임상 시나리오 검증이 필수입니다.
6. `TRA01/TRA02 사용자·운영자 지침서`
   운영 이후 수용률과 안정성에 직접 영향이 큽니다.

## 7. 교육용으로 설명할 때 좋은 흐름

표준산출물 목록서를 기반으로 설명할 때는 아래 순서가 가장 이해하기 쉽습니다.

1. `REQ02 사용자 요구사항 정의서`
   무엇을 왜 만들었는지
2. `ANA01 업무정의서`
   현행 업무와 개선 업무가 어떻게 다른지
3. `ANA02 유스케이스 명세서`
   사용자 행동 중심으로 시스템을 설명
4. `ANA06 / DES07 인터페이스 정의서`
   데이터가 어디서 들어오는지 설명
5. `DES06 사용자 인터페이스 설계서`
   실제 사용자 화면 설명
6. `DES09~DES12 시험 관련 문서`
   검증 구조 설명
7. `TRA01~TRA07 이행 문서`
   오픈과 운영까지 연결

## 8. 한 줄 요약

이 표준산출물 체계로 CDSS를 폭포수 방식으로 정리하면,
`요구사항과 업무흐름을 먼저 고정하고 -> 데이터/인터페이스/화면/규칙을 설계한 뒤 -> 시험과 운영전환 문서까지 단계적으로 완성하는 방식`으로 이해하면 됩니다.
